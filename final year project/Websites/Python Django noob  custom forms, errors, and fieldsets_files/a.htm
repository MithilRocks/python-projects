        I think the approach you landed on makes sense. The downside, of course, is that you've written a bunch of python code that describes presentation logic. If you want to keep templates non-codey, what you want is a way to specify fieldsets (and other surrounding markup) in a template, while specifying the available fields, validation logic, and submission processing logic in python.

        I sketched an idea for dealing with this about a year ago, but haven't implemented it (yet). One client project of mine has a heavily customized version of this idea. Here's the idea. In python, you do something like:

        class UserRegisterForm(TemplateForm):
        username = forms.CharField()
        password = forms.CharField(widget=forms.PasswordInput)
        password_confirm = forms.CharField(widget=forms.PasswordInput)

        class Meta:
        template = "forms/userRegisterForm.html"

        And then you write forms/userRegisterForm.html using some special tags:

        {% load templateform %}
        {% templateform %}
        <h2>Register an Account</h2>
        {% form_errors %}
        {% form_field "username" %}
        <fieldset><legend>Password</legend>
        {% form_field "password" %}
        {% form_field "password_confirm" %}
        </fieldset>
        <input type="submit" value="Submit">
        {% endtemplateform %}

        The idea is that each template tag will default to the current behavior of form rendering, with the option of using extra parameters or sub-tags to customize things. So if you just wrote {% templateform %}{% endtemplateform %}, you'd simply get a default rendering of the form, with form errors, then hidden fields, then named fields. If you put a {% form_field %} tag inside of that, that field will get rendered first. If you want to put {% form_errors %} in a special place, you can do that, all from the template.

                                          mikepk Mod robertf57 • a year ago

                                          I am by no means a Django expert so you should take my opinions with a grain of salt :)

                                          You can add validation logic in the clean method, but I think that's generally a bad idea. In my opinion, it's better to have explicit validators. When you read the username input, you can see the validation required for that field, rather than hunting through the logic in the clean method.

                                          It's one of the things I don't like about Django validators, that a multi-input validation is impossible without monkeying with 'clean' (although this may be different in newer versions since I wrote this). Messing with clean seems more like a hack. Its generally concerning when you have to use two different mechansims to provide validation.
